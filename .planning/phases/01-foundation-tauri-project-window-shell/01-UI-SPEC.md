## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Task text (compact) | 17px | 700 (bold) | 1.3 |
| Empty state prompt | 17px | 600 (semibold) | 1.3 |
| Label / secondary | 13px | 400 (regular) | 1.4 |

Phase 1 uses exactly 2 weights: **700** for the primary task text and **600** for the empty state prompt. Weight 400 is declared for Phase 2 use (labels, secondary text) but is not rendered in Phase 1.

Font sizes: 3 declared (13px, 17px). The 17px size is chosen as the midpoint of the REQUIREMENTS.md spec of "16-18px" — optimized for readability at arm's length on Retina displays while fitting comfortably in the 300px-wide compact shell with 16px horizontal padding on each side (268px usable text width).

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `rgba(30, 30, 30, var(--widget-opacity, 0.95))` | Widget background — dark charcoal base |
| Secondary (30%) | `#FF6B2B` | Orange accent fill — the visual identity of the widget |
| Accent (10%) | `#FFFFFF` with 0.95 opacity | Text on orange background |
| Completion green | `#34C759` | Task completion state (token established now, used in Phase 2) |

### Detailed color application for Phase 1

**Widget background strategy:** The widget uses a dark charcoal base (`rgba(30, 30, 30)`) as the dominant surface, with `#FF6B2B` orange as the strong visual identity color. Two valid approaches for the compact shell:

- **Option A (recommended):** Solid `#FF6B2B` orange background for the entire compact widget. White text directly on orange. This matches the "confident badge" metaphor from the PRD.
- **Option B:** Dark charcoal background with a strong orange left border (4px) or top accent bar. White text on dark.

**Phase 1 implements Option A:** The compact widget shell is a solid orange (`#FF6B2B`) rounded rectangle with white text. This is the "badge on your screen" identity.

| Element | Background | Text/Foreground |
|---------|-----------|-----------------|
| Compact widget body | `#FF6B2B` | `#FFFFFF` (opacity 0.95) |
| Widget shadow | n/a | `0 4px 24px rgba(0, 0, 0, 0.3)` |
| Transparent window canvas | `transparent` | n/a |

**Opacity:** Widget opacity is controlled via CSS custom property `--widget-opacity` applied to the outermost `.widget` container. Default: `0.95`. Range: `0.3` to `1.0`. In Phase 1, hardcode `0.95`; the slider arrives in Phase 2.

**Contrast check:** `#FFFFFF` on `#FF6B2B` yields a contrast ratio of approximately 3.0:1. This is below WCAG AA for normal text (4.5:1) but acceptable because: (a) the text is 17px bold (large text threshold is 18.66px bold or 24px normal — 17px bold is borderline), (b) this is a personal-use desktop widget, not a public web page, (c) the white-on-orange is the deliberate brand identity. If contrast needs improvement, use `#1E1E1E` dark text on orange instead — but the PRD explicitly specifies white/near-white text on orange.

Source: REQUIREMENTS.md DESIGN-01, DESIGN-02, DESIGN-04; key_design_decisions in prompt; ARCHITECTURE.md CSS section

## Window Geometry

| Property | Compact Mode (Phase 1) | Expanded Mode (Phase 2, tokens only) |
|----------|----------------------|--------------------------------------|
| Width | 300px (LogicalSize) | 400px (LogicalSize) |
| Height | 60px (LogicalSize) | 200px (LogicalSize) |
| Border radius | 12px | 12px |
| Box shadow | `0 4px 24px rgba(0, 0, 0, 0.3)` | `0 4px 24px rgba(0, 0, 0, 0.3)` |

**Shadow implementation:** Native window shadow is disabled (`shadow: false` in tauri.conf.json). CSS `box-shadow` on the `.widget` div provides the float effect. This requires the transparent window canvas to be slightly larger than the visible widget to accommodate the shadow spread. Add 24px padding to the window dimensions: actual window size = `348px x 108px` to give the shadow room to render without clipping.

**Correction on window size:** The `width: 300` and `height: 60` in tauri.conf.json define the webview area. The shadow extends beyond the widget div. To prevent shadow clipping, either:
1. Set window size to 348x108 and add 24px of transparent padding around the `.widget` div, OR
2. Set window size to 300x60 and accept shadow clipping at edges

**Recommendation:** Use approach 1 (larger window with transparent padding). The extra transparent area is invisible and non-interactive (clicks pass through transparent pixels). The visible widget remains 300x60.

Source: ARCHITECTURE.md window config; key_design_decisions; PITFALLS.md transparent window section

## Interaction Contract

### Drag behavior (Phase 1 scope)

| Interaction | Behavior |
|-------------|----------|
| Click and drag anywhere on widget | Window moves with cursor |
| Quick click (mousedown + mouseup, no movement) | No action in Phase 1 (expand arrives in Phase 2) |
| Right-click | No action in Phase 1 |

**Implementation:** Apply `data-tauri-drag-region` attribute to the `.widget` container div. The entire compact widget surface is the drag region.

**Click vs drag disambiguation:** In Phase 1, the widget only drags — there is no expand action yet. The `data-tauri-drag-region` attribute handles native drag. Click disambiguation logic (movement threshold of 3-5px within 200ms) should be implemented in Phase 1 to prepare for Phase 2's expand-on-click, but the click handler is a no-op.

Source: ARCHITECTURE.md drag regions; PITFALLS.md drag-vs-click; REQUIREMENTS.md WIN-06, WIN-07

### Position persistence (Phase 1 scope)

| Event | Action |
|-------|--------|
| Window moved (drag ends) | Save `{x, y}` to tauri-plugin-store, debounced 500ms |
| App launch | Read stored position, validate against monitor bounds, call `setPosition()` |
| Stored position is off-screen | Fall back to top-right corner of primary display, 12px margin from edges |

Source: REQUIREMENTS.md WIN-08, WIN-09; ARCHITECTURE.md position persistence

## Phase 1 Visual States

The widget has exactly 2 visual states in Phase 1:

### State 1: Empty (default on first launch)

- This state can be demonstrated with a hardcoded string; actual task entry is Phase 2
